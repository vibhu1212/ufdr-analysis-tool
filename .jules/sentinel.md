## 2024-05-24 - [Avoid `eval()` on external metadata]
**Vulnerability:** Found `eval()` being used in `media/video_processor.py` to parse video frame rates from FFmpeg probe output (`video_stream.get('r_frame_rate', '0/1')`). This is a critical Remote Code Execution (RCE) vulnerability because an attacker could craft a malicious video file with embedded Python code in its metadata, which would be executed when the system runs `ffmpeg.probe()`.
**Learning:** External data from tools like FFmpeg should be treated as untrusted input. The assumption that video metadata will always be safely formatted as `"num/den"` is insecure.
**Prevention:** Never use `eval()` to parse strings. Use safe parsing methods such as string splitting (`str.split()`), regex, and explicit type conversion (`float()`, `int()`).

## 2024-05-24 - [Enforce Allowlist for SQLite Table Names]
**Vulnerability:** In `frontend/components/ufdr_upload_component.py`, dynamically retrieved table names from uploaded SQLite databases were passed directly to `cursor.execute(f"PRAGMA table_info({table})")` and `SELECT * FROM {table}`. Although `sqlite_master` natively enforces rules, a maliciously constructed UFDR file containing arbitrary or crafted table names could be uploaded to trigger SQL injection vectors if those names contain characters that break out of the statement, since `f-strings` are not parameterization.
**Learning:** You cannot parameterize table names in PRAGMA or dynamic SELECT statements in SQLite (`?`). The only safe approach when handling unverified database files is to strict match or use a predefined allowlist.
**Prevention:** Always validate dynamically extracted table or column names against an application-level allowlist (e.g. `SQLValidator.ALLOWED_TABLES`) or enforce strict alphanumeric constraints prior to formatting them into SQL statements.
